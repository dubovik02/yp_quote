## Реализованный функционал

# Сервер стриминга котировок.
1. Генерация данных.
2. TCP-сервер. Приложение запускает TCP-сервер, который принимает команды от клиентов.
3. Протокол команд:
STREAM <host>:<udp_port> <tcker1>,<ticker2>...<tickerN> | <"*">
<"*"> - все тикеры.

Пример команды: STREAM 127.0.0.1:34254 AAPL,TSLA.

4. Обработка команды.
Cервер отправляет сгенерированные данные (только для запрошенных тикеров) на указанный UDP-адрес.

5. Механизм Ping/Pong (Keep-Alive):
Если в течение заданного тайм-аута от клиента не приходит "Ping", сервер останавливает стриминг данных для клиента и завершает соответствующий поток.

6. Многопоточность. Каждый новый UDP-стрим для клиента и основной генератор данных работают в отдельных потоках.

# Клиентский модуль стриминга котировок.
1. Интерфейс. Приложение предоставляет способ указать:
Адрес и порт TCP-сервера
Порт для приёма UDP-данных
Путь к файлу со списком тикеров для подписки. Если файл тикеров не указан - запрос отправляется на все тикеры.
Реализовано с помощью крейта clap.


2. Формат файла тикеров. Тикеры перечисляются по одному на строку. 

3. Клиент подключается к TCP-серверу и отправляет команду STREAM с нужными параметрами.

4. Приём UDP-данных. После отправки команды клиент:
Создает UDP-сокет для приёма данных на указанный в команде порт.
В основном потоке начинает получать и выводить в консоль приходящие котировки.

5.  Параллельно клиент  периодически отправляет "Ping" сообщение на UDP-адрес сервера — на тот же адрес и порт, с которого приходят данные.
